CLI-first incident analysis

Understand production incidents from logs in seconds

Sydes analyzes raw logs and generates a structured root-cause hypothesis, timeline, and evidence - without manually scanning thousands of lines.

Get started → View docs →
$ sydes incident analyze logs/*.log \
  --query "Why is checkout returning 503 errors?"

→ API Gateway timeouts to db-proxy causing 503 errors on checkout requests

/* why this is useful */
When incidents happen, engineers are forced to scan thousands of lines across multiple services, hosts, and timestamps. The problem is not lack of data - it is lack of structure. Sydes reconstructs what happened from logs so failures can be investigated as evidence, not raw text.
/* how it works */
  • 1. parse and interpret raw logs
  • 2. group related failure signals into events and clusters
  • 3. build a timeline and generate a root-cause hypothesis
/* query vs grep */
  • --grep filters logs by exact patterns
  • --query lets you investigate a suspected issue in plain English
  • best results usually come from combining both
$ sydes incident analyze logs/*.log \
  --query "Why are session lookups failing?" \
  --grep "session lookup failed"

→ cache_unavailable causing session lookup failures in auth-service
