CLI incident analysis

Understand production incidents
from logs

Sydes turns raw logs into incident structure you can investigate.

/* what incident analysis does */
  • reconstructs incident timelines from logs
  • identifies likely root causes
  • surfaces supporting evidence
  • connects failure signals across services
/* quickstart */
$ pip install sydes
$ export OPENAI_API_KEY=your_key
/* query + grep */

Ask a focused question and narrow to a concrete signal.

$ sydes incident analyze logs/*.log \
  --model openai:gpt-4.1-mini \
  --require-model \
  --query "Why is checkout failing?" \
  --grep "timeout|checkout"
/* multi-service logs */

Reconstruct the failure path across services.

$ sydes incident analyze api.log worker.log gateway.log \
  --model openai:gpt-4.1-mini \
  --require-model \
  --query "Why is checkout failing?"
/* time window */

Narrow the incident window before reasoning.

$ sydes incident analyze logs/*.log \
  --model openai:gpt-4.1-mini \
  --require-model \
  --query "Why is checkout failing?" \
  --from 2026-04-03T15:05:00 \
  --to 2026-04-03T15:10:00

/* why this is useful */
When incidents happen, engineers are forced to scan thousands of lines across multiple services, hosts, and timestamps. The problem is not lack of data - it is lack of structure. Sydes reconstructs what happened from logs so failures can be investigated as evidence, not raw text.
/* how it works */
  • 1. parse and interpret raw logs
  • 2. group related failure signals into events and clusters
  • 3. build a timeline and generate a root-cause hypothesis
/* why this exists */
Logs are raw data. Incidents are structure.
Sydes reconstructs what happened so engineers investigate systems, not text.
